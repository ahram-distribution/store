# 23 – Architecture Governance

**التصنيف:** حوكمة معمارية  
**الغرض:** تحديد قواعد الحوكمة المعمارية — كيف تُدار التغييرات، كيف تُفرض القواعد، كيف تُحل الخلافات  
**تاريخ الإصدار:** 5 يوليو 2026  
**الإصدار:** 1.0  
**الحالة:** ✅ نافذ

---

## 1. المبادئ

1. **العمارة هي القانون** — لا تنفيذ يخالف العمارة المعتمدة
2. **كل تغيير معماري يحتاج ADR** — لا قرارات شفوية
3. **الشفافية** — كل ADR موثق ومتاح للجميع
4. **السرعة مع الحوكمة** — ADR لا يعني بيروقراطية — ADR يمكن أن يكون صفحة واحدة

---

## 2. هرم القرارات المعمارية

```
┌──────────────────────────────────────────────────────┐
│     المستوى 1: Architecture Decision Record (ADR)    │
│     أي تغيير في الطبقات، القدرات، التبعيات، ADRs     │
├──────────────────────────────────────────────────────┤
│     المستوى 2: تنفيذ معتمد (Approved Implementation) │
│     تحقيق العمارة — Providers, Services, Mappers      │
├──────────────────────────────────────────────────────┤
│     المستوى 3: قرارات تنفيذية (Implementation Choice) │
│     اسم متغير، ترتيب دوال، خيارات تنسيق — لا ADR     │
└──────────────────────────────────────────────────────┘
```

### متى نحتاج ADR؟

| نعم — ADR مطلوب | لا — مجرد قرار تنفيذي |
|-----------------|----------------------|
| إضافة قدرة أعمال جديدة | اسم ملف داخل Application Service |
| إضافة Provider Interface جديد | ترتيب الـ Parameters في دالة |
| تغيير قواعد التبعية | اختيار مكتبة (ضمن القائمة المسموحة) |
| إضافة طبقة جديدة | تنسيق الكود (ESLint) |
| تغيير استراتيجية Aggregates | إضافة Validator جديد |
| إضافة سياق نشر جديد (Desktop/Mirror) | إضافة Mapper جديد |
| تغيير استراتيجية المعاملات | **أي شيء لا يغير العمارة** |

---

## 3. دورة حياة ADR

```
[مقترح] ──► [مراجعة] ──► [معتمد / مرفوض]
    │                        │
    │                        ├── [معتمد] → يُنفذ + تُحدّث الوثائق
    │                        └── [مرفوض] → يُغلق مع التوضيح
    │
    └── [مسحوب] ← المقدِّم يسحب الاقتراح
```

### 3.1 القالب الرسمي للـ ADR

```
# ADR-NNN: عنوان القرار

| الحقل | القيمة |
|-------|--------|
| الرقم | ADR-NNN |
| التاريخ | YYYY-MM-DD |
| المقدِّم | (الاسم) |
| الحالة | مقترح / معتمد / مرفوض / مسحوب |

## السياق (Context)
(وصف المشكلة أو الظرف)

## المشكلة (Problem)
(ما المشكلة التي نحلها؟)

## البدائل (Alternatives)
(2-3 بدائل مع إيجابيات وسلبيات)

## القرار (Decision)
(ما اخترنا ولماذا)

## التأثير (Consequences)
(التأثير الإيجابي والسلبي + الملفات التي ستتغير)
```

### 3.2 صلاحيات الاعتماد

| المستوى | من يوافق |
|---------|---------|
| تغيير في الطبقات أو القدرات | Product Owner |
| إضافة Provider Interface | Architecture Lead + Product Owner |
| تغيير استراتيجية (مثل Saga→Transaction) | Architecture Lead + Product Owner |
| ADR عاجل (تأمين، أداء) | Architecture Lead (يُبلّغ PO) |

## 4. قواعد الحوكمة الصارمة (Hard Governance Rules)

| الرقم | القاعدة | المسؤول عن التطبيق |
|-------|---------|-------------------|
| **G-01** | لا استدعاء `supabase.rpc()` من أي Page أو Component | Code Review |
| **G-02** | لا استيراد من `providers/implementations` في `application/` | Code Review + ESLint |
| **G-03** | لا استيراد من `lib/supabase` في `domain/` | Code Review + ESLint |
| **G-04** | كل Provider ينفذ Interface واحد على الأقل من `providers/contracts/` | Code Review |
| **G-05** | كل Application Service يستقبل Providers عبر Constructor | Code Review |
| **G-06** | لا Business Rule في UI | Code Review |
| **G-07** | الـ Domain لا يستورد أي مكتبة خارجية | Code Review + ESLint |
| **G-08** | كل ADR جديد يُسجل في 20_ARCHITECTURE_DECISIONS.md | Architecture Lead |
| **G-09** | لا يُحذف `services/` القديم قبل التأكد من عدم استخدامه | Architecture Lead |

---

## 5. آليات فرض الحوكمة (Enforcement Mechanisms)

| الآلية | الوصف | الأولوية |
|--------|-------|---------|
| **Code Review** | كل PR يُراجع للتأكد من الإلتزام بالعمارة | إلزامي |
| **ESLint Rules** | قواعد `import/no-restricted-paths` تمنع الاستيرادات الممنوعة | تلقائي |
| **Dependency Cruiser** | رسم بياني للتبعيات — يُفحص دورياً | شهري |
| **Architecture Review** | مراجعة معماري دورية (كل 3 أشهر) | ربع سنوي |
| **Violation Log** | سجل الخروقات المعمارية — للتحسين المستمر | مستمر |

### 5.1 قواعد ESLint المقترحة

```json
{
  "rules": {
    "import/no-restricted-paths": ["error", {
      "zones": [
        { "target": "src/pages", "from": "src/providers" },
        { "target": "src/pages", "from": "src/lib/supabase.ts" },
        { "target": "src/application", "from": "src/providers/implementations" },
        { "target": "src/domain", "from": "src/providers" },
        { "target": "src/domain", "from": "src/lib/supabase.ts" },
        { "target": "src/domain", "from": "src/application" },
        { "target": "src/providers/implementations", "from": "src/application" }
      ]
    }]
  }
}
```

## 6. معالجة الخروقات (Violation Handling)

| مستوى الخطورة | المثال | الإجراء |
|--------------|--------|---------|
| **Critical** | استدعاء `supabase.rpc()` من Page | رفض PR + إصلاح فوري |
| **Major** | Domain يستورد من Supabase | رفض PR + إصلاح |
| **Minor** | استيراد غير صحيح لكن لا ضرر فوري | ملاحظة في PR + إصلاح |
| **Warning** | نمط غير متوافق (مستقبلاً) | تسجيل + مراجعة في المراجعة الدورية |

### 6.1appeals

إذا اختلف المطور مع القرار المعماري:
1. يناقش مع Architecture Lead
2. إذا لم يُحل → ADR جديد
3. إذا لم يُحل → Product Owner يحسم

---

## 7. مسؤوليات الحوكمة

| الدور | المسؤولية |
|-------|-----------|
| **Architecture Lead** | الحفاظ على العمارة، الموافقة على ADRs، المراجعات الدورية |
| **Product Owner** | الموافقة على ADRs التي تؤثر في قدرات الأعمال، تحديد الأولويات |
| **Developer** | الإلتزام بالعمارة، الإبلاغ عن الخروقات، اقتراح ADRs للتطوير |
| **Code Reviewer** | التحقق من الإلتزام بقواعد التبعية والأنماط المعمارية |

---

## 8. جدول المراجعات الدورية

| المراجعة | التكرار | المحتوى |
|----------|---------|---------|
| **مراجعة التبعيات** | شهري | Dependency Cruiser — هل هناك خروقات؟ |
| **مراجعة ADRs** | ربع سنوي | هل الـ ADRs لا تزال صالحة؟ |
| **مراجعة العمارة الكاملة** | سنوي | هل العمارة لا تزال تناسب احتياجات النظام؟ |
