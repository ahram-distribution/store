# 13 – Provider Migration Plan

**التصنيف:** تصميم معماري — خطة الترحيل من Services الحالية إلى Data Provider Architecture  
**الغرض:** توثيق خطة تدريجية لترحيل 165+ نقطة وصول بيانات من الشيفرة الحالية إلى النظام الجديد  
**تاريخ الإصدار:** 5 يوليو 2026  
**الإصدار:** 1.0  
**الحالة:** مسودة للاعتماد

---

## 1. المبادئ

1. **لا تغيير جذري (Big Bang)** — الترحيل تدريجي، خطوة بخطوة
2. **النظام القديم والجديد يعملان معاً** —共存 مؤقتاً
3. **كل خطوة قابلة للعكس** — إذا فشلت خطوة، نتراجع دون تأثير
4. **كل خطوة تنتج شيئاً قابلاً للاختبار** — لا ترحيل دون تحقق
5. **الأولوية للـ Providers الأكثر استخداماً** — Order, Product, Customer أولاً

---

## 2. استراتيجية Strangler Fig (التسلق التدريجي)

```
الوقت ──────────────────────────────────────────────►

┌────────────────────────────────────────────────────┐
│ النظام القديم (services/)                    ●●●●●●│
│ النظام الجديد (providers/ + application/)          │
│              ○○○○○○●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●│
│                                                    │
│ المرحلة 1    المرحلة 2    المرحلة 3    المرحلة 4  │
│ (Foundation) (Core)       (Expansion)  (Completion)│
└────────────────────────────────────────────────────┘

● = يعمل   ○ = تحت التطوير
```

**المبدأ:** كل صفحة جديدة أو تحديث يستخدم النظام الجديد. النظام القديم يبقى حتى تختفي كل مرجعية له.

## 3. المراحل

### المرحلة 0: Foundation (تقدير: 5 أيام)

**الهدف:** بناء الهيكل الأساسي — المجلدات، الـ Contracts، Registry، أول Provider (Mock).

| المهمة | المخرجات | التقدير |
|--------|----------|---------|
| إنشاء مجلدات `providers/`, `application/`, `domain/` | الهيكل الفارغ | 0.5 يوم |
| إنشاء واجهات `IProvider`, `ICrudProvider`, `IReadOnlyProvider` | 3 ملفات | 0.5 يوم |
| إنشاء `IProviderRegistry` وتنفيذه (`ProviderRegistry.ts`) | 2 ملفات | 0.5 يوم |
| إنشاء `ProviderException` ومشتقاته | 1 ملف | 0.25 يوم |
| إنشاء `MockDataStore.ts` — بيانات وهمية في الذاكرة | 1 ملف | 0.5 يوم |
| تنفيذ **Mock Provider** لـ Order (كامل مع CRUD + دوال الحالة) | 1 ملف | 1 يوم |
| إنشاء `src/bootstrap.ts` — تسجيل Mock Provider | 1 ملف | 0.25 يوم |
| كتابة اختبار وحدة لـ MockOrderProvider | 1 ملف | 0.5 يوم |
| استدعاء `bootstrap.ts` في `main.tsx` | تعديل بسيط | 0.25 يوم |

**مجموع المرحلة 0: ~5 أيام**

**حالة النظام في النهاية:**
- ✅ MockProvider يعمل ويُسجل في bootstrap
- ✅ أي صفحة يمكنها استخدام `registry.resolve<IOrderProvider>('order')` وستحصل على بيانات وهمية
- ✅ النظام القديم لا يزال يعمل كما هو

| ترحيل RPCs من `services/auth.ts` إلى `SupabaseUserProvider` | ترحيل ~5 استدعاء | 1 يوم |
| إنشاء `SupabaseUnitOfWork` — تنفيذ بسيط (PostgreSQL transactions) | 1 ملف | 1 يوم |
| إنشاء `IUnitOfWork` Interface | 1 ملف | 0.25 يوم |

| إنشاء Mappers لـ Order, Product, Customer | 3 ملفات | 0.5 يوم |

**مجموع المرحلة 1: ~8 أيام**

**حالة النظام في النهاية:**
- ✅ SupabaseProviders لـ 4 كيانات رئيسية

- ✅ النظام القديم لا يزال يعمل — **لا ترحيل للصفحات بعد**

| اختبار شامل للصفحة المترحلة (مقارنة النتائج مع النظام القديم) | 1 يوم |
| ترحيل CreditService بقية — `SupabaseCreditProvider` | ترحيل 20 RPC | 2 يوم |
| ترحيل AttendanceService — `SupabaseAttendanceProvider` | ترحيل 28 RPC | 2.5 يوم |
| إنشاء Application Services لدعم Credit و Attendance | 3-4 ملفات | 1 يوم |
| ترحيل صفحة الـ Attendance إلى النظام الجديد | 1 صفحة | 1 يوم |
| كتابة اختبارات A/B: هل يعطي النظام الجديد نفس نتائج القديم؟ | بعد كل ترحيل | 1 يوم |

**مجموع المرحلة 2: ~10 أيام**

**حالة النظام في النهاية:**
- ✅ صفحتان تستخدمان النظام الجديد بالكامل
- ✅ بقية الصفحات لا تزال تستخدم services/القديمة
- ✅ نظام A/B للمقارنة

### المرحلة 3: Expansion (تقدير: 12 أيام)

**الهدف:** ترحيل جميع الـ Services المتبقية وكل الصفحات.

| المهمة | المخرجات | التقدير |

| إنشاء Application Services متبقية | 8-10 ملفات | 2 يوم |
| ترحيل كل صفحة لاستخدام Application Services (42 صفحة) | 42 صفحة × 2 ساعة | 3 أيام |
| ترحيل `engine/pricing.ts` إلى `domain/services/PricingService.ts` | نقل + تحسين | 0.5 يوم |
| ترحيل `hooks/useCapability.ts` لاستخدام `IUserProvider` بدل RPC مباشر | 1 ملف | 0.5 يوم |
| ترحيل `store/cart.ts` لاستخدام PricingService الجديد | 1 ملف | 0.5 يوم |

## 7. خطة الطوارئ

| السيناريو | الإجراء |
|-----------|---------|
| خطأ في Provider جديد | الرجوع إلى النظام القديم — لا حذف للـ services القديمة حتى اكتمال الترحيل |
| تأخير في مرحلة | تقليل نطاق المرحلة (ترحيل أقل، اختبار أكثر) |
| عدم رضا المستخدم | إيقاف الترحيل والعودة للنظام القديم — تحليل الأسباب |
| تغيير Supabase Schema | تحديث الـ Provider فقط — لا تأثير على Application Layer |
