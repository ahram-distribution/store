**النسخة:** 1.0  
**التصنيف:** وثيقة معمارية وتنفيذية رسمية  
**تاريخ الإصدار:** 5 يوليو 2026

---

## 1. Executive Summary

Ahram ERP هو نظام تخطيط موارد مؤسسية تشغيلي لشركة الأهرام للتجارة والتوزيع. ليس موقعاً تعريفياً، بل منصة متكاملة تدير قوائم الأسعار، المبيعات، الزيارات الميدانية، التحصيلات، المستهدفات، وأداء مندوبي المبيعات.

النظام مبني بهدف تملك البيانات بالكامل، مع القدرة على تشغيله كموقع ويب (SPA) أو كبرنامج Windows EXE مستقل. لا يرتبط بمزود قاعدة بيانات محدد، بل يعبر طبقة تجريد (Data Provider Layer) تسمح باستبدال مصدر البيانات (Supabase، PostgreSQL محلي، SQL Server) دون إعادة كتابة واجهة المستخدم.

---

## 2. Product Vision

- **البيانات تحت سيطرة الشركة:** لا تبعية لطرف خارجي في الوصول إلى البيانات أو تشغيل النظام.
- **النظام يعمل Web وWindows:** نفس الكود، نفس القواعد، واجهتان مختلفتان.
- **مصدر البيانات قابل للاستبدال:** طبقة Data Provider تفصل الواجهة عن قاعدة البيانات.
- **نسخة Windows ليست مجرد غلاف:** هي مركز إدارة وتحكم كامل مع إمكانية التشغيل المحلي دون اتصال بالإنترنت.

---

## 3. Team Operating Model

- **Product Owner:** هو المسؤول الوحيد عن القرارات الوظيفية وأولويات التطوير والمتطلبات. يمتلك رؤية المنتج ويصدق على كل تغيير.
- **Architecture / Review:** مسؤول تحليلي يراجع القرارات التقنية، يصادق على سلامة المعمارية، ويكتشف التعارضات قبل التنفيذ.
- **Implementation Engine (OpenCode):** المسؤول عن التنفيذ وفق القرارات المعتمدة فقط، ولا يتخذ قرارات معمارية أو وظيفية دون الرجوع إلى Product Owner.

**آلية العمل:**
1. Product Owner يقرر المتطلبات.
2. Architecture/Review يضع الخطة التقنية أو يراجعها.
3. OpenCode ينفذ.
4. المراجعة والاختبار قبل النشر.

## 4. System Architecture

### 4.1 Presentation Layer
طبقة واجهة المستخدم (React + TypeScript + Tailwind CSS)، مسؤولة فقط عن عرض البيانات والتقاط إجراءات المستخدم. لا تحتوي على منطق أعمال مباشر. تدعم الاتجاه RTL بالكامل.

### 4.2 Data Provider Layer
طبقة تجريد بين الواجهة ومصدر البيانات. الواجهة لا تستدعي Supabase مباشرة، بل تتحدث مع Data Provider. هذا يسمح بتبديل مصدر البيانات (Supabase, PostgreSQL محلي, SQL Server) دون تغيير سطر واحد في الـ UI.

### 4.3 Business Rules Layer
منطق الأعمال المركزي: حسابات الأسعار، المستهدفات، تقييم الأداء، KPI. هذه الطبقة قابلة للاختبار بشكل مستقل عن الواجهة وعن قاعدة البيانات.

### 4.4 Database Layer
قاعدة البيانات العلائقية. حالياً Supabase (PostgreSQL مستضاف). مستقبلاً PostgreSQL محلي أو أي RDBMS يدعمه Data Provider.

### 4.5 Desktop Layer (Windows EXE)
تطبيق Windows مستقل (Electron أو Tauri) يشغل نفس كود الويب، مع إمكانية الوصول إلى قاعدة بيانات محلية (PostgreSQL أو SQLite للتخزين المؤقت) ودعم العمل دون اتصال.

### 4.6 Sync / Mirror Layer
طبقة المزامنة بين القاعدة البعيدة (Supabase) والقاعدة المحلية. مسؤولة عن نقل التغييرات في الاتجاهين، مع حل تعارضات إن وجدت.

### 4.7 Backup / Restore Layer
طبقة النسخ الاحتياطي والاستعادة. تعمل على مستوى قاعدة البيانات وليس على مستوى الملفات. تدعم الاستعادة الانتقائية للبيانات.

---

## 5. Deployment Philosophy

- المشروع يجب أن يعمل على Supabase حالياً دون تغيير.
- المشروع يجب أن يعمل على Local PostgreSQL دون إعادة كتابة الواجهة.
- الواجهة لا ترتبط مباشرة بمصدر البيانات — أي تغيير في قاعدة البيانات يتطلب فقط تحديث Data Provider.
- نسخة Windows EXE تشغل نفس كود الويب مع إضافة إمكانيات محلية.
- البنية التحتية قابلة للتوسع: من مزود مستضاف إلى محلي بالكامل والعكس.

---

## 6. Data Ownership Strategy

الهدف الأول والأخير: **امتلاك البيانات محلياً**.

- **Operational Mirror:** نسخة تشغيلية كاملة من قاعدة البيانات تعمل محلياً (PostgreSQL). هذه النسخة ليست احتياطية فقط، بل جاهزة للتشغيل الفوري إذا توقف المصدر البعيد.
- **Backup:** نسخة احتياطية للطوارئ. غير قابلة للتشغيل المباشر. تحتاج استعادة قبل الاستخدام.

**لماذا النسخ الاحتياطي وحده غير كافٍ:**
- الاستعادة تستغرق وقتاً.
- لا يمكن العمل على النسخة الاحتياطية مباشرة.
- الـ Operational Mirror يضمن استمرارية العمل دون انقطاع.

**لماذا يجب أن يكون لدينا مصدر محلي قابل للتشغيل:**
- قطع الإنترنت لا يوقف العمل.
- سرعة الوصول للبيانات (Localhost أسرع من أي API).
- سيطرة كاملة دون قيود مزود الخدمة.

## 7. Desktop Strategy

### لماذا Windows EXE
- وجود تطبيق مستقل يضمن استمرارية العمل دون الاعتماد على المتصفح أو الاتصال بالإنترنت.
- يمكن الوصول إلى موارد النظام المحلية (طابعات، ملفات، قاعدة بيانات محلية).

### ما الذي ستقدمه نسخة الويندوز
- نفس واجهة الويب + إمكانيات إضافية محلية.
- وصول مباشر لقاعدة بيانات محلية.
- مزامنة خلفية تلقائية مع الخادم البعيد.
- أدوات إدارة وتحكم مركزية.

### مشاركة الكود
- 100% من كود الواجهة يُشارك بين Web وDesktop.
- الـ Desktop Layer تضيف فقط: التحكم في النافذة، الوصول لنظام الملفات، إدارة الاتصال بقاعدة البيانات المحلية.

### Offline / Online
- العمل دون اتصال: البيانات من القاعدة المحلية.
- العمل مع اتصال: مزامنة تلقائية أو اتصال مباشر بالخادم البعيد.
- الانتقال بين الحالتين تلقائي دون فقدان بيانات.

### الاتصال بقاعدة البيانات المحلية
- Desktop EXE يتصل مباشرة بـ Local PostgreSQL عبر TCP.
- لا حاجة لـ REST API أو HTTP — اتصال قاعدة بيانات مباشر.

## 8. Business Domains

### A) Business Activity
ما فعله الموظف فعلياً خلال الفترة بغض النظر عن المستهدفات. تشمل:

- **Submitted Orders:** طلبات تم إرسالها فعلياً.
- **Sales Created:** مبيعات تم إنشاؤها في النظام.
- **Completed Visits:** زيارات مكتملة (بدأت وانتهت).
- **Collections Received:** تحصيلات تم استلامها فعلياً.
- **Customers Created:** عملاء جدد تمت إضافتهم.

### B) Target Achievement
ما يُحتسب على الهدف بعد تطبيق قواعد الأعمال. تشمل:

- **Delivered Orders:** الطلبات المسلمة فعلياً وليست المرسلة فقط.
- **Delivered Sales:** صافي المبيعات بعد خصم المرتجعات المعتمدة.
- **Target Customers:** عدد العملاء المحتسب ضمن الهدف حسب قواعد التوزيع.
- **Target Collections:** التحصيلات المحتسبة ضمن الهدف حسب القرار المعتمد.

### C) Field Activity
السلوك الميداني — ليس إنتاجاً بل انضباطاً:

- الحضور والانصراف
- ساعات العمل
- فترات الاستراحة
- المسافة المقطوعة
- نقاط التتبع
- أيام العمل الفعلية

---

## 9. KPI Contract

**القواعد الحاسمة:**
- كل KPI له Business Event واحد. لا يوجد KPI يجيب على سؤالين مختلفين.
- كل KPI له Canonical Source واحد (قاعدة بيانات، جدول، RPC). لا يوجد حسابان مختلفان لنفس الرقم.
- لا توجد حسابات مكررة — أي KPI يُحسب في مكان واحد فقط.
- لا يوجد KPI بمعنيين مختلفين — إذا اختلف المعنى فهو KPI مختلف وله اسم مختلف.

### الـ KPIs المعتمدة:

| الاسم | التعريف | المصدر |
|-------|----------|--------|
| **Visits** | عدد الزيارات التي بدأها المندوب | نظام التتبع |
| **Orders** | عدد الطلبات المسلمة | جدول الطلبات |
| **Sales** | صافي قيمة المبيعات (بعد المرتجعات) | جدول المبيعات |
| **Customers** | عدد العملاء المكلف بهم | جدول العملاء |
| **Collections** | قيمة التحصيلات المستلمة | جدول التحصيلات |
| **Working Days** | عدد أيام العمل الفعلية | نظام التتبع |
| **Net Hours** | ساعات العمل الفعلية (بعد خصم الاستراحات) | نظام التتبع |
| **Break Time** | إجمالي وقت الاستراحة | نظام التتبع |
| **Distance** | المسافة المقطوعة خلال يوم العمل | نظام التتبع |
| **Tracking Points** | عدد نقاط التتبع المسجلة | نظام التتبع |

---

## 10. Canonical Naming Rules

- **إذا اختلف المعنى التجاري، اختلف الاسم.**
- **Activity Sales** تختلف عن **Target Sales**.
- **Activity Orders** تختلف عن **Target Orders**.
- **Activity Customers** تختلف عن **Target Customers**.
- لا يجوز استخدام نفس الاسم لرقمين معنويين مختلفين في النظام.
- كل اسم يعبر عن Business Meaning واضح واحد فقط.

---

## 11. Reporting Philosophy

- التقارير موجهة للإدارة التنفيذية.
- المدير لا يهتم بـ RPC أو تفاصيل تقنية — يهتم بمعنى الرقم.
- كل رقم في التقرير يجب أن يكون:
  - قابلًا للتتبع إلى مصدره (Drill-down).
  - مفهوم المعنى التجاري (Sales = المبيعات بعد المرتجعات، وليس إجمالي الفواتير).
  - غير قابل للتأويل.
- التقارير تجيب على ثلاثة أسئلة:
  1. ماذا فعل الموظف؟ (Business Activity)
  2. ماذا احتسب على الهدف؟ (Target Achievement)
  3. كيف كان الأداء الميداني؟ (Field Activity)

## 12. Current Status

### تم الإنجاز والمعتمد:
- قاعدة الكود: React + TypeScript + Vite + Tailwind CSS — SPA.
- قاعدة البيانات: Supabase PostgreSQL مع RPC للاستعلامات.
- المصادقة: Supabase Auth مع التحكم بالأدوار (الإدارة العليا، مدير بيع، مندوب مبيعات).
- شاشة قوائم الأسعار (Sales List): عرض جدولي مع تحميل PDF.
- تحميل PDF عبر `html-to-image` (SVG foreignObject) للحفاظ على الحروف العربية.
- طباعة PDF تحترم الفلتر الحالي (الشركة المحددة أو الكل).
- شاشة تقارير المدير (Manager Reports).
- Service Worker للتخزين المؤقت وتحديثات SW.
- أسماء مقننة: `sales-list` في شاشة العنوان.
- `.nojekyll` في البنية لضمان تشغيل GitHub Pages.
- GitHub Actions CI/CD للنشر على GitHub Pages.

### في الطريق:
- طبقة Data Provider لفصل الواجهة عن قاعدة البيانات.
- إعادة هيكلة Business Rules Layer.

### ما زال في مرحلة التخطيط:
- نسخة Windows EXE.
- Local PostgreSQL / Mirror.
- Sync / Offline / Online.
- Control Center / Admin Tools.
- النسخ الاحتياطي والاستعادة.

### ما تقرر عدم فعله الآن:
- إعادة كتابة المشروع من الصفر.
- التحويل إلى Microsoft Access.
- استخدام SQLite كقاعدة بيانات رئيسية.
- أي حل مؤقت يضرب المعمارية (مثل ربط الواجهة مباشرة بقاعدة البيانات دون Data Provider).
- أي تعديل يكرر حساب KPI من أكثر من مصدر.

## 13. Roadmap

### المرحلة الحالية — Web SPA
- الواجهة الحالية تعمل على Supabase + GitHub Pages.
- استكمال شاشات التقارير المتبقية.
- تطبيق طبقة Data Provider.

### المرحلة التالية — Windows EXE
- تطوير تطبيق Windows (Electron/Tauri) يشغل نفس كود الويب.
- إضافة إمكانية الاتصال بقاعدة بيانات محلية.
- مزامنة أساسية بين المحلي والبعيد.

### المرحلة التالية — Local PostgreSQL / Mirror
- إنشاء نسخة تشغيلية محلية من PostgreSQL.
- تكوين الاتصال بين Desktop EXE وقاعدة البيانات المحلية.
- هيكلة Data Provider لدعم قواعد بيانات متعددة.

### المرحلة التالية — Sync / Offline / Backup
- مزامنة ثنائية الاتجاه بين Supabase والمحلي.

- طبقة Backup/Restore.

### المرحلة التالية — Control Center / Admin Tools
- أدوات إدارة المستخدمين والأدوار.
- لوحة تحكم مركزية.
- أدوات تشخيص ومراقبة الأداء.

## 14. Non-Goals

ما لا نريد فعله في أي مرحلة حالية:

- **إعادة كتابة المشروع من الصفر:** الكود الحالي هو الأساس للتوسع، ليس للاستبدال.
- **التحويل إلى Access:** Access ليس بديلاً مناسباً لنظام ERP تشغيلي.
- **استخدام SQLite كقاعدة رئيسية:** SQLite للتخزين المؤقت فقط، وليس للبيانات التشغيلية الأساسية.
- **حلول مؤقتة تضرب المعمارية:** أي حل يربط الواجهة مباشرة بقاعدة البيانات متجاوزاً Data Provider مرفوض حتى لو كان أسرع في التنفيذ.
- **تكرار حسابات KPI:** كل KPI يُحسب في مكان واحد فقط، وتستهلكه كل الشاشات والتقارير من نفس المصدر.

---

## 15. Governance

- **أي تغيير جديد** يجب أن يمر عبر:
  1. Product Owner يقرر الحاجة.
  2. مراجعة أثر التغيير على المعمارية والقواعد الحالية.
  3. اعتماد الخطة التنفيذية قبل كتابة الكود.
- **من يقرر:** Product Owner فقط.
- **ما يجب مراجعته قبل التنفيذ:**
  - هل التغيير يحترم طبقة Data Provider؟
  - هل يكرر حساب KPI موجود؟
  - هل يضيف مصدر بيانات جديد بدون طبقة تجريد؟
  - هل يؤثر على إمكانية التشغيل المحلي (Local Mirror)؟
- **أي ميزة جديدة** يجب أن تحترم نفس المعمارية الحالية.

---

## 16. Success Criteria

Ahram ERP يعتبر ناجحاً عندما:

1. **البيانات تحت السيطرة:** الشركة تملك نسخة تشغيلية محلية من بياناتها.
2. **النظام يعمل Web وWindows:** نفس الوظائف، واجهتان.
3. **التقارير دقيقة:** كل رقم في التقرير له مصدر واحد وتعريف واحد.
4. **KPI موحد:** لا يوجد خلاف على معنى أي مؤشر أداء.
5. **التوسع المستقبلي سهل:** إضافة شاشة جديدة أو مصدر بيانات جديد لا يتطلب إعادة كتابة.
6. **النظام قابل للنقل:** التبديل بين Supabase و PostgreSQL المحلي يتم بتغيير إعداد فقط (بفضل Data Provider).
7. **استمرارية التشغيل:** قطع الإنترنت لا يوقف العمل بفضل الـ Local Mirror.
