## 1) نموذج Work Mode المُقترح

### المشكلة

`ATTENDANCE_REDESIGN_IMPACT_ANALYSIS.md` استخدمت 4 modes: `fixed_shift`, `flexible_hours`, `hourly`, `executive`.

المشكلة: `executive` ليس Work Mode — إنه رتبة تنظيمية (Role/Rank). الإدارة العليا قد تعمل دوام ثابت أو ساعات مرنة.

### الحل: نموذج ثنائي البعد (Two-Dimensional Policy)

```
employee_work_policies
│
├── work_location (مكان العمل)
│   ├── field     ← ميداني (يتنقل، يزور عملاء)
│   └── office    ← مكتبي (موقع ثابت)
│
├── schedule_type (نظام الدوام)
│   ├── fixed_shift   ← دوام ثابت (09:00 - 17:00)
│   ├── flexible      ← دوام مرن (عدد ساعات محدد بدون توقيت)
│   └── hourly        ← بالساعة (احتساب دقيق للأجر)
│
├── tracking_required (منطقي)
├── required_daily_hours
├── shift_start / shift_end (لـ fixed_shift)
└── grace_minutes
```

### لماذا بعدين بدلاً من 4 modes؟

| الموديل القديم | المشكلة |
|----------------|---------|
| `fixed_shift` | يصلح لميداني ومكتبي — لكن الفرق في التتبع |
| `flexible_hours` | يصلح لميداني ومكتبي — نفس الشيء |
| `hourly` | يصلح لميداني ومكتبي — الفرق في مكان العمل فقط |
| `executive` | ليس Work Mode — مدير قد يكون ميداني أو مكتبي |

### التركيبات الممكنة

| work_location | schedule_type | مثال | tracking_required |
|--------------|---------------|------|-------------------|
| field | fixed_shift | مندوب مبيعات دوام 8-4 | true |
| field | flexible | مندوب مبيعات ساعات مرنة | true |
| field | hourly | سائق/عامل ميداني بالساعة | true |
| office | fixed_shift | موظف إدارة دوام 9-5 | false |
| office | flexible | موظف إدارة ساعات مرنة | false |
| office | hourly | موظف إدارة بالساعة (نادر) | false |

### هيكل الجدول المقترح

```sql
CREATE TABLE IF NOT EXISTS public.employee_work_policies (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    employee_id uuid NOT NULL REFERENCES public.employees(id),

    -- البعد الأول: مكان العمل
    work_location varchar(20) NOT NULL DEFAULT 'field'
        CHECK (work_location IN ('field', 'office')),

    -- البعد الثاني: نظام الدوام
    schedule_type varchar(20) NOT NULL DEFAULT 'flexible'
        CHECK (schedule_type IN ('fixed_shift', 'flexible', 'hourly')),

    -- التتبع
    tracking_required boolean NOT NULL DEFAULT true,

    -- ساعات العمل
    required_daily_hours decimal(4,2) NOT NULL DEFAULT 8.00,
    shift_start time,              -- مهم لـ fixed_shift فقط
    shift_end time,                -- مهم لـ fixed_shift فقط
    grace_minutes integer NOT NULL DEFAULT 0,

    -- Metadata
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX uq_emp_work_policy ON public.employee_work_policies (employee_id);
```

### كيف يعمل النظام مع 3 أنواع Work؟

#### دوام ثابت (fixed_shift)

```
السلوك:
  - وقت البدء: يُقارن بـ shift_start. إذا تأخر > grace_minutes → is_late = true
  - وقت النهاية: يُقارن بـ shift_end. إذا أنهى قبلها → early_departure = true
  - الغياب: إذا لم يسجل start بحلول time > shift_start + grace_minutes → absent

الحسابات:
  - net_hours = ended_at - started_at - breaks                               ← مشترك
  - is_late = started_at > shift_start + grace_minutes                       ← خاص
  - late_minutes = started_at - (shift_start + grace_minutes)                ← خاص
  - early_departure = ended_at < shift_end                                    ← خاص
  - hours_deficit = max(0, required_daily_hours - net_hours)                 ← مشترك

المؤشرات المهمة:
  - الالتزام بالدوام (حضر في الوقت؟) — الأهم
  - صافي ساعات العمل (أنجز المطلوب؟)
  - الإنصراف المبكر (غادر قبل الوقت؟)

مثال:
  دوام: 09:00 - 17:00, grace: 15 دقيقة
  - بدأ 09:10 → on time (ضمن السماح)
  - بدأ 09:20 → is_late = true, late_minutes = 5
  - أنهى 16:30 → early_departure = true (غادر قبل 17:00 بـ 30 دقيقة)
  - أنهى 17:10 → normal
  - net_hours = 8:00 - breaks = 7:30 مثلاً
```

#### دوام مرن (flexible)

```
السلوك:
  - وقت البدء: لا يُقارن — يُسجل فقط (للتقرير، وليس للتقييم)
  - وقت النهاية: لا يُقارن — يُسجل فقط
  - المهم الوحيد: هل net_hours >= required_daily_hours؟
  - لا يوجد late / early / absent (بالمفهوم التقليدي)
  - الغياب: فقط إذا لم يسجل start في اليوم → absent

الحسابات:
  - net_hours = ended_at - started_at - breaks                               ← مشترك
  - hours_deficit = max(0, required_daily_hours - net_hours)                 ← مشترك

المؤشرات المهمة:
  - صافي ساعات العمل — المؤشر الوحيد المهم
  - هل حقق الـ required_daily_hours؟

مثال:
  دوام مطلوب: 8 ساعات
  - بدأ 10:00 → لا مشكلة
  - بدأ 13:00 → لا مشكلة (طالما سيعمل 8 ساعات)
  - أنهى 19:00 → net_hours = 8:00 - 0:30 breaks = 7:30 → hours_deficit = 0:30
  - التقييم: لم يحقق المطلوب (7:30 < 8:00)
```

#### بالساعة (hourly)

```
السلوك:
  - وقت البدء: يُسجل فقط (للحساب المالي)
  - وقت النهاية: يُسجل فقط (للحساب المالي)
  - لا يوجد late / early — لأن الموظف يُحتسب له كل دقيقة يعملها

الحسابات:
  - net_hours = ended_at - started_at - breaks                               ← مشترك
  - (لا يوجد required_daily_hours — كل ساعة تحسب)

المؤشرات المهمة:
  - net_hours الدقيقة — المؤشر الوحيد (للمحاسبة)
  - عدد أيام العمل
  - (لا ينطبق Target vs Actual بنفس الطريقة)

مثال:
  - بدأ 08:00 → سجل البداية
  - استراحة 12:00-12:30 → خصمت 0:30
  - أنهى 16:00 → net_hours = 8:00 - 0:30 = 7:30
  - يحاسب على 7.5 ساعة
```

### مقارنة الأنواع الثلاثة

| الجانب | دوام ثابت | دوام مرن | بالساعة |
|--------|-----------|----------|---------|
| يقارن وقت البدء؟ | ✅ نعم | ❌ لا | ❌ لا |
| يقارن وقت النهاية؟ | ✅ نعم | ❌ لا | ❌ لا |
| يُحتسب تأخير؟ | ✅ نعم | ❌ لا | ❌ لا |
| يُحتسب انصراف مبكر؟ | ✅ نعم | ❌ لا | ❌ لا |
| مهم: هل حقق الساعات؟ | ✅ مهم | ✅ **الأهم** | ✅ يُحتسب فقط |
| net_hours مشترك؟ | ✅ نعم | ✅ نعم | ✅ نعم |
| Target vs Actual | ✅ ينطبق | ✅ ينطبق | ❌ محدود

### تأثير work_location على النظام

| الجانب | field | office |
|--------|-------|--------|
| GPS Tracking | مطلوب (افتراضياً) | غير مطلوب (افتراضياً) |
| الخريطة | يظهر في Live Map | لا يظهر |
| المسافة | تُحتسب | 0 / null |
| نقاط التتبع | تُجمع | لا تُجمع |
| المسار اليومي | يُعرض في DayMap | لا يُعرض |
| حالة الاتصال | connected/delayed/lost | not_applicable |

### هل tracking_required منفصل عن work_location؟

نعم. `tracking_required` يمكن أن يتجاوز القيمة الافتراضية.

**مثال:** موظف field لكنه في تدريب ولا يحتاج تتبع هذا الشهر → `tracking_required = false` مع بقاء `work_location = 'field'`.

**مثال عكسي:** موظف office لكنه في مهمة ميدانية مؤقتة → `tracking_required = true` مع بقاء `work_location = 'office'`.

### الفرق بين هذا التصميم والتصميم السابق

| الجانب | التصميم السابق (4 modes) | التصميم الجديد (بعدين) |
|---------|--------------------------|------------------------|
| Executive | موجود كـ mode 🚫 | يُحذف — يُعامل كـ Role |
| Fixed Shift | موجود ✅ | موجود كـ schedule_type |
| Flexible Hours | موجود ✅ | موجود كـ schedule_type |
| Hourly | موجود ✅ | موجود كـ schedule_type |
| Field/Office | غير موجود | موجود كـ work_location |
| المرونة | أقل — 4 خيارات فقط | أعلى — 6 تركيبات |
| الوضوح | يخلط بين rank و mode | يفصل rank عن mode |

## 2) Tracking Policy

### المبدأ

`tracking_required` هو حقل لكل موظف في `employee_work_policies`.

القيمة الافتراضية:
- `field` → `tracking_required = true`
- `office` → `tracking_required = false`

لكن يمكن تجاوزها فردياً.

### ماذا يحدث عندما tracking_required = false؟

**Tracking Engine:**
```
قبل بدء trackingEngine.start():
  ← قراءة employee_work_policies WHERE employee_id = current_user
  ← إذا tracking_required = false:
       • لا تبدأ setInterval
       • لا تفتح IndexedDB
       • engine.isRunning = false
       • نقطة start تُرسل من start_workday RPC
       • نقطة end تُرسل من end_workday RPC
  ← نعم:
       • Tracking يبدأ كالمعتاد
```

**في AttendancePage:**
```
  ← tracking_required = false:
       • لا يُظهر "آخر موقع"
       • لا يُظهر المسافة
       • لا يُظهر عدد نقاط التتبع
       • لا يُظهر حالة الاتصال
       • زر "عرض على الخريطة" → رسالة "لا يتطلب تتبع"
```

**في LiveMonitoringPage:**
```
get_live_workday_overview:
  ← tracking_required = false:
       • connection_status = 'not_applicable'
       • لا يُحتسب ضمن connection_loss_count
       • لا يظهر في last_points
       • يظهر في قائمة الموظفين لكن بدون أيقونة موقع
       • يظهر في عداد "مكتب/غير مطلوب تتبع"
  ← tracking_required = true:
       • كل شيء كالمعتاد
```

**في TeamMapPage / الخريطة:**
```
  ← tracking_required = false:
       • لا يظهر على الخريطة نهائياً
  ← tracking_required = true:
       • يظهر بنقطة (حسب الحالة)
```

**في EmployeeDayMapPage:**
```
  ← tracking_required = false للموظف المطلوب:
       • route فارغ (0 نقاط tracking)
       • مسافة = 0 / null
       • رسالة واضحة: "هذا الموظف لا يتطلب تتبع"
       • عرض فقط: session info + timeline + الإنتاجية
       • إخفاء الخريطة بالكامل
  ← tracking_required = true:
       • كل شيء كالمعتاد
```

**في التقارير:**
```
get_workday_report:
  ← tracking_required = false:
       • distance_meters = null
       • tracking_point_count = null
       • connection_status = 'not_applicable'
       • بقية المؤشرات (ساعات، طلبات، زيارات) كالمعتاد
```

## 5) Target vs Actual

### الوضع الحالي

النظام لديه بالفعل:

| الجدول | الوظيفة |
|--------|---------|
| `employee_monthly_targets` | أهداف شهرية لكل موظف: sales, visits, orders, new_customers, collections |
| `performance_weights_config` | أوزان التقييم السنوية |
| `employee_weight_overrides` | تجاوز أوزان فردي |

و RPC `get_governed_target_performance` يحسب:
- Achievement percentage لكل KPI
- Weighted overall score
- Best/weakest employee

### المشكلة

هذه الميزة موجودة ولكنها **غير مرتبطة** بشاشات الحضور — موجودة في نظام منفصل (Targets Module).

الحل: دمج Target vs Actual في **كل شاشات الحضور**.

### كيف يحسب الـ Target اليومي؟

```
Monthly Target (من employee_monthly_targets)
  ÷
أيام العمل المتوقعة في الشهر (عادة 22-26)
  =
Daily Target
```

**مثال:**
```
الزيارات: 90 / شهر → 90 ÷ 22 = 4.09/يوم (نقرب: 4 زيارات/يوم)
الطلبات: 60 / شهر → 60 ÷ 22 = 2.73/يوم (نقرب: 3 طلبات/يوم)
المبيعات: 180,000 / شهر → 180,000 ÷ 22 = 8,182/يوم
التحصيل: 50,000 / شهر → 50,000 ÷ 22 = 2,273/يوم
عملاء جدد: 20 / شهر → 20 ÷ 22 = 0.91/يوم (نقرب: 1/يوم)
```

### أين يظهر Target vs Actual؟

| الشاشة | ماذا يظهر؟ |
|--------|------------|
| **AttendancePage** (للمندوب) | أهدافه اليومية مقابل منجزه الفعلي — شريط تقدم لكل KPI |
| **FieldOperationsCenter** (للمشرف) | Target vs Actual للفريق ككل — عدادات عليا |
| **ReportsPage** | Target vs Actual مفصل لكل موظف + إجمالي الفريق + نسبة الإنجاز |
| **EmployeeDayMapPage** | Target vs Actual لليوم المحدد بجانب الـ KPI cards |
| **EmployeeProfilePage** | Target vs Actual شهري لكل موظف |

### آلية الاحتساب

```
لكل موظف في اليوم:
  actual_orders = COUNT(*) WHERE session_id = X
  actual_visits = COUNT(*) WHERE session_id = X AND check_out_at IS NOT NULL
  actual_sales = SUM(order_total) WHERE session_id = X
  actual_collections = SUM(collection_amount) WHERE session_id = X
  actual_new_customers = COUNT(DISTINCT customer_id WHERE first_order_date = today)

  target_orders = employee_monthly_targets.orders_target / working_days_in_month
  target_visits = employee_monthly_targets.visits_target / working_days_in_month
  ...إلخ

  achievement_pct = actual / target * 100
```

### متى لا يظهر Target vs Actual؟

1. الموظف ليس لديه target للشهر الحالي → رسالة "لا توجد أهداف محددة لهذا الشهر"
2. work_location = office + schedule_type = hourly → قد لا ينطبق Target vs Actual (يعتمد على طبيعة العمل المكتبي)
3. الفترة الزمنية المحددة لا تحتوي على target (مثل: آخر 7 أيام في غير الشهر الحالي) → يُظهر فقط الـ Actual بدون مقارنة

### Work Hours Ledger (دفتر ساعات العمل)

عنصر أساسي في التصميم — ليس مجرد مؤشر عابر، بل دفتر يومي متكامل لساعات العمل.

#### المكونات

| المؤشر | المصدر | ينطبق على |
|--------|--------|-----------|
| **ساعات كل يوم** (Daily Hours) | `workday_sessions.duration` - `workday_breaks.duration` | الكل |
| **إجمالي ساعات الشهر** (Monthly Total) | SUM(sessions) - SUM(breaks) للشهر | الكل |
| **متوسط ساعات العمل** (Daily Average) | Monthly Total ÷ أيام العمل في الشهر | الكل |
| **أفضل يوم** (Best Day) | أعلى `net_hours` في الشهر | الكل |
| **أقل يوم** (Worst Day) | أدنى `net_hours` في الشهر (باستثناء أيام الغياب) | الكل |
| **مقارنة مع الشهر السابق** | الفرق بين Current Month Total - Previous Month Total | الكل |

#### Net Work Hours = المؤشر الرئيسي المشترك

**Net Work Hours هو المؤشر الوحيد الذي ينطبق على كل أنواع العمل الثلاثة:**

| النوع | هل net_hours ينطبق؟ | ملاحظات |
|-------|--------------------|---------|
| **دوام ثابت** (fixed_shift) | ✅ نعم — `ended_at - started_at - breaks` | يُقارن بـ `required_daily_hours` للكشف عن التأخير والانصراف المبكر |
| **دوام مرن** (flexible) | ✅ نعم — `ended_at - started_at - breaks` | المقياس الأساسي: هل أنجز `required_daily_hours`؟ |
| **بالساعة** (hourly) | ✅ نعم — `ended_at - started_at - breaks` | الأدق — يُستخدم للمحاسبة المالية |

**لماذا هو المؤشر الرئيسي؟**
- ينطبق على كل الموظفين بغض النظر عن work_location أو schedule_type
- يقيس الالتزام الأساسي: هل حضر الموظف وأنجز ساعاته؟
- أساسي لكل التقارير: يومي، أسبوعي، شهري
- أساس المقارنة بين الموظفين والفرق والأشهر

#### كيف يُحتسب Net Hours؟

```sql
-- لكل session:
net_hours = EXTRACT(EPOCH FROM (ended_at - started_at)) / 3600
            - COALESCE(break_duration, 0)
-- break_duration = مجموع durations من workday_breaks المرتبطة بالـ session
```

## 7) Best / Worst Day

### المشكلة

التحليل السابق استخدم "best/worst day" بناءً على composite score واحد فقط.

### الحل: مؤشرات متعددة مع Composite Score

#### المكونات

| المكون | الوزن الافتراضي | المصدر | ينطبق على |
|--------|----------------|--------|-----------|
| صافي ساعات العمل | 30% | workday_sessions - workday_breaks | field, office |
| الطلبات (Orders) | 25% | orders table | field |
| الزيارات (Visits) | 20% | visits table | field |
| التحصيلات (Collections) | 15% | collections table | field |
| العملاء الجدد | 10% | orders.first_order | field |

**ملاحظة:** الأوزان للموظف المكتبي (office) تختلف — لا ينطبق عليه الطلبات/الزيارات/التحصيلات → يعتمد فقط على ساعات العمل + مؤشرات مكتبية (مثل: مهام مكتملة).

#### طريقة الحساب

```
لكل يوم:
  net_hours_score = (actual_net_hours / required_daily_hours) × 100  (مقتطعة عند 100)
  orders_score = (actual_orders / daily_target_orders) × 100  (مقتطعة عند 100)
  visits_score = (actual_visits / daily_target_visits) × 100  (مقتطعة عند 100)
  collections_score = (actual_collections / daily_target_collections) × 100  (مقتطعة عند 100)
  new_customers_score = (actual_new_customers / daily_target_new_customers) × 100  (مقتطعة عند 100)

  composite_score =
    net_hours_score × 0.30 +
    orders_score × 0.25 +
    visits_score × 0.20 +
    collections_score × 0.15 +
    new_customers_score × 0.10

  النتيجة: رقم بين 0 و 100
```

**مثال:**
```
يوم 15/06:
  ساعات: 8:40 من 8:00 → 100% (مقتطعة) × 0.30 = 30
  طلبات: 5 من 4 → 100% (مقتطعة) × 0.25 = 25
  زيارات: 8 من 6 → 100% (مقتطعة) × 0.20 = 20
  تحصيل: 3,000 من 2,500 → 100% (مقتطعة) × 0.15 = 15
  عملاء جدد: 1 من 1 → 100% × 0.10 = 10
  Composite Score = 100/100 ✅
```

#### أين يظهر Best/Worst Day؟

| الشاشة | ماذا يظهر؟ |
|--------|------------|
| **EmployeeProfilePage** (تبويب الحضور) | أفضل يوم + أسوأ يوم في الشهر مع composite score |
| **ReportsPage** | يمكن فرز الموظفين حسب composite score |
| **AlertsPage** | إذا composite score < 30 → تنبيه "أداء منخفض" |

### آلية العمل

**القرار:** **ديناميكي (محسوب) فقط** — لا حاجة لجدول `operational_alerts` في المرحلة الحالية.

| المزايا | العيوب |
|---------|--------|
| ✅ لا حاجة لجدول جديد — توفير في تعقيد القاعدة | ❌ لا يمكن تسجيل من ألغلق التنبيه أو متى |
| ✅ لا stale data — التنبيهات تُحتسب فوراً من الوضع الحالي | ❌ لا يمكن تتبع تاريخ التنبيهات (متى ظهر أولاً) |
| ✅ بسيط — يحتاج فقط تحديث RPC `get_alerts` الموجود | ❌ لا يمكن أرشفة التنبيهات المُغلقة |
| ✅ لا يحتاج cron job أو trigger | ❌ التنبيهات "لحظية" — إذا زال السبب يختفي التنبيه فوراً |

**التوصية:** ديناميكي. إذا احتجنا لاحقاً تتبع تاريخ التنبيهات — نضيف جدول `operational_alerts` في المرحلة التالية (Future Roadmap).

### متى تُحتسب التنبيهات؟

```
1. عند طلب AlertsPage → تُحتسب latest alerts ديناميكياً من البيانات الحالية
2. في FieldOperationsCenter → تُحتسب مع get_live_workday_overview مباشرة
3. في AttendancePage → تُحتسب مع get_my_workday_status مباشرة
```

### كيف يُحتسب كل تنبيه؟

```
NO_START:
  WHERE DATE(today) AND workday_sessions.started_at IS NULL
    AND schedule_type = 'fixed_shift'
    AND CURRENT_TIME > shift_start + grace_minutes

NO_ORDERS:
  WHERE DATE(today) AND workday_sessions IS NOT NULL
    AND work_location = 'field'
    AND AGE(CURRENT_TIME, started_at) > '3 hours'
    AND orders.count WHERE session_id = X = 0

NO_VISITS:
  (مثل NO_ORDERS لكن مع visits.count = 0)

NO_COLLECTIONS:
  WHERE workday_sessions.ended_at IS NOT NULL (أنهى اليوم)
    AND collections.count WHERE session_id = X = 0

TRACKING_LOST:
  WHERE tracking_required = true
    AND AGE(CURRENT_TIME, last_tracking_point.created_at) > '15 minutes'
    AND workday_sessions.started_at IS NOT NULL
    AND workday_sessions.ended_at IS NULL

LOW_HOURS:
  WHERE schedule_type IN ('fixed_shift', 'flexible')
    AND net_hours < required_daily_hours
    AND CURRENT_TIME > shift_end - '1 hour'  (لـ fixed_shift)
    OR CURRENT_TIME > '16:00' (قرب نهاية اليوم لـ flexible)

LONG_STOP:
  WHERE tracking_required = true
    AND AGE(CURRENT_TIME, last_two_points -> same location > 30 min)
    AND NOT is_on_break

BREAK_TIMEOUT:
  WHERE workday_breaks.ended_at IS NULL
    AND AGE(CURRENT_TIME, workday_breaks.started_at) > workday_settings.max_break_duration

ABSENCE:
  WHERE NOT EXISTS (SELECT 1 FROM workday_sessions WHERE DATE = TODAY)
    AND CURRENT_TIME > '12:00'
    AND employee.is_active = true

LOW_PERFORMANCE:
  WHERE composite_score < 30% لآخر 3 أيام عمل
```

### كيف تظهر في الشاشات المختلفة؟

| الشاشة | ماذا يظهر |
|--------|-----------|
| **FieldOperationsCenter** | آخر 3-5 تنبيهات في شريط سفلي + عداد "⚠️ X تنبيه نشط" |
| **AlertsPage** | كل التنبيهات مع فلاتر + قسم نشط + قسم مغلق |

## 16) سياسة الاحتفاظ بالبيانات (Data Retention)

### المبدأ العام

**أرشفة Tracking Points — دراسة مستقبلية (Future Study).** لا تُنفَّذ حالياً.

**السبب:**
- قاعدة البيانات حالياً صغيرة — عدد الموظفين محدود
- `tracking_points` بحجمها الحالي لا تؤثر على الأداء (بضع مئات من النقاط/يوم/موظف)
- لا داعي لتعقيد التنظيف والأرشفة في المرحلة الأولى
- يمكن اتخاذ قرار الأرشفة لاحقاً مرتكزاً على حجم حقيقي (وليس تخمينياً)

**الموقف الحالي:**
- `tracking_points` تبقى **كاملة إلى الأبد** في المرحلة الحالية
- `auto_cleanup_tracking_data` موجود ويُشغل — لكنه لا يفعل شيئاً (أو يُعدَّل ليكون no-op)
- المسافة المقطوعة محفوظة في `workday_sessions.distance_meters` (محسوبة من `get_employee_day_map` عند الطلب)

### ما الذي يبقى بالتفصيل إلى الأبد؟

| البيانات | المدة | السبب |
|----------|-------|-------|
| `workday_sessions` | للأبد | سجل حضور أساسي — لا يمكن حذفه |
| `workday_breaks` | للأبد | مرتبط بـ sessions — يبقى معها |
| `employee_work_policies` | للأبد | سجل سياسات العمل — تاريخي |
| `tracking_points` | للأبد (حالياً) | بيانات التتبع — تُراجع لاحقاً |
| `orders` | للأبد | بيانات مبيعات — خارج نطاق الحضور |
| `visits` | للأبد | بيانات زيارات — خارج نطاق الحضور |
| `collections` | للأبد | بيانات تحصيل — خارج نطاق الحضور |
| `employees` | للأبد | بيانات الموظفين — أساسية |

### ما الذي لا يجوز حذفه؟

| البيانات | لماذا؟ |
|----------|--------|
| `workday_sessions` | أساس التقارير وسجل الحضور |
| `workday_breaks` | مرتبطة بـ sessions — بدونها سجل الاستراحات ناقص |

### دراسة مستقبلية: أرشفة tracking_points

**متى ننفذها؟** عندما يصل حجم `tracking_points` إلى > 1 مليون صف (أو يبدأ الأداء في التأثر).

**الخيارات المطروحة (للمستقبل):**

| الخيار | الوصف |
|--------|-------|
| A — تلخيص (نقطة/5 دقائق) | تحويل النقاط الخام إلى نقاط ملخصة كل 5 دقائق — توفير ~97% مع بقاء Route شبه دقيق |
| B — نقل إلى أرشيف | جدول `tracking_points_archive` منفصل — البيانات تبقى كاملة لكن في جدول بارد (نادر الاستعلام) |
| C — حذف بعد 45 يوم | حذف tracking_points الأقدم من 45 يوم — الأكثر توفيراً لكن لا يمكن الرجوع للنقاط القديمة |

**التوصية:** عند الحاجة — الخيار B (أرشيف) هو الأكثر أماناً لأن البيانات تبقى متاحة للاستعلام عند الحاجة.

## 17) تصميم Route Map الكامل

### المصادر الحالية

النظام لديه حالياً `get_employee_day_map` (V2) في `20260611_phase6_attendance_v2_complete.sql`.

هذا RPC موجود ويعمل:

| المكون | المصدر | الوصف |
|--------|--------|-------|
| نقاط التتبع | `tracking_points` | كل نقاط GPS للموظف في اليوم المحدد |
| نقاط البداية | أول نقطة في `tracking_points` (point_type = 'start') | دائرة خضراء على الخريطة |
| نقاط النهاية | آخر نقطة (point_type = 'end') أو آخر نقطة GPS | دائرة حمراء على الخريطة |
| نقاط الزيارات | `visits` table (مع check_in_at في نفس اليوم) | ماركر مع أيقونة |
| المسافة | Haversine بين النقاط المتتالية | محسوبة في الـ RPC |
| التوقفات الطويلة | نقاط متتالية ضمن ~50m لأكثر من 30 دقيقة | محسوبة في الـ RPC |

### كيف يُرسم المسار حالياً؟

```

2. يرجع JSON مع:

     "employee_id": "...",
     "date": "2026-06-15",

       { lat, lng, captured_at, point_type },

     "stops": [                    // التوقفات الطويلة

3. Frontend:
```

### كيف تُحتسب المسافة؟

```sql
-- Haversine Formula (موجود في get_employee_day_map)
-- تقريب المسافة بين نقطتين GPS
-- لكل نقطتين متتاليتين: calculate المسافة
-- المجموع = المسافة الإجمالية

WITH ordered_points AS (
    SELECT lat, lng, captured_at,
        ROW_NUMBER() OVER (ORDER BY captured_at) AS rn
    FROM tracking_points
    WHERE session_id = p_session_id
    ORDER BY captured_at
)
SELECT SUM(
    6371000 * 2 * ASIN(SQRT(
        SIN(RADIANS(lat - prev_lat)/2)^2 +
        COS(RADIANS(lat)) * COS(RADIANS(prev_lat)) *
        SIN(RADIANS(lng - prev_lng)/2)^2
    ))
) AS total_distance_meters
FROM (
    SELECT
        lat, lng,
        LAG(lat) OVER (ORDER BY captured_at) AS prev_lat,
        LAG(lng) OVER (ORDER BY captured_at) AS prev_lng
    FROM ordered_points
) sub
WHERE prev_lat IS NOT NULL;
```

### كيف تُكتشف التوقفات الطويلة؟

```sql
-- Halle: نقاط متتالية ضمن ~50m لأكثر من 30 دقيقة
-- 1. احسب المسافة بين كل نقطة وسابقتها
-- 2. جمّع النقاط متتالية التي المسافة بينها < 50m
-- 3. إذا وقت المجموعة > 30 دقيقة ← هذه وقفة طويلة

WITH point_groups AS (
    SELECT lat, lng, captured_at,
        SUM(CASE WHEN distance_from_prev > 50 THEN 1 ELSE 0 END)
            OVER (ORDER BY captured_at) AS group_id
    FROM (
        SELECT lat, lng, captured_at,
            COALESCE(haversine(
                LAG(lat) OVER (ORDER BY captured_at),
                LAG(lng) OVER (ORDER BY captured_at),
                lat, lng
            ), 0) AS distance_from_prev
        FROM tracking_points
        WHERE session_id = p_session_id
        ORDER BY captured_at
    ) sub
)
SELECT
    MIN(lat) AS lat, MIN(lng) AS lng,
    MIN(captured_at) AS started_at,
    MAX(captured_at) AS ended_at,
    EXTRACT(EPOCH FROM MAX(captured_at) - MIN(captured_at)) / 60 AS duration_minutes
FROM point_groups
GROUP BY group_id
HAVING EXTRACT(EPOCH FROM MAX(captured_at) - MIN(captured_at)) / 60 >= 30;
```

### آلية العمل الكاملة

### السيناريوهات

#### 1) انقطع الإنترنت

```
1. Timer ينطلق → GPS يلتقط نقطة → تُحفظ في IndexedDB
2. محاولة sync_tracking_points → فشل (No Internet)
3. النقطة تبقى في IndexedDB
4. كل 30 ثانية: نفس السيناريو — تتراكم النقاط
```

#### 2) عاد الإنترنت بعد 10 دقائق

```
1. Timer ينطلق → GPS يلتقط نقطة → تُحفظ في IndexedDB
2. محاولة sync_tracking_points → نجاح (Internet عاد)

4. الرد: success → تُحذف النقاط من IndexedDB
5. Timer التالي: نفس الآلية — Queue فارغ → يُرسل النقطة الحالية فقط
```

#### 3) عاد الإنترنت بعد ساعة

#### 4) عاد الإنترنت بعد عدة ساعات

```
1. ~500-1000 نقطة في Queue (حسب 8 ساعات × 120/ساعة)
2. محاولة الإرسال: قد تكون دفعة كبيرة لكن مقبولة (JSON ~50KB)
3. إذا كانت الدفعة كبيرة جداً → تقسيم إلى دفعات 100 نقطة
4. إرسال متسلسل (أو متوازي) حتى تفريغ الـ Queue
```

### Batch Sending Logic

```typescript
async function flushQueue(): Promise<void> {
  const allPoints = await getAllQueuedPoints(); // من IndexedDB
  if (allPoints.length === 0) return;

  const BATCH_SIZE = 100;
  for (let i = 0; i < allPoints.length; i += BATCH_SIZE) {
    const batch = allPoints.slice(i, i + BATCH_SIZE);
    try {
      const { error } = await supabase.rpc('sync_tracking_points', {
        p_token: token,
        p_points: batch.map(p => ({
          p_session_id: p.session_id,
          p_lat: p.lat,
          p_lng: p.lng,
          p_accuracy: p.accuracy,
          p_captured_at: p.captured_at,
          p_point_type: p.point_type,
        })),
      });
      if (error) throw error;
      // حذف النقاط المرسلة
      await deleteQueuedPoints(batch.map(p => p.id));
    } catch (err) {
      // وقف الإرسال — المحاولة القادمة
      break;
    }
  }
}
```

### ماذا لو أغلق المستخدم المتصفح أثناء Queue غير مرسلة؟

```
→ IndexedDB متصلة بالـ Browser Storage
→ النقاط تبقى في IndexedDB حتى بعد إغلاق التطبيق
→ عند فتح التطبيق لاحقاً: Restore Runtime → flush Queue
→ أنظر Section 20: Runtime Restore
```

### ضمان عدم فقدان النقاط — ملخص

| السيناريو | هل تُفقد نقاط؟ | الحل |
|-----------|---------------|------|
| انقطع الإنترنت | ❌ لا | IndexedDB يحفظ النقاط حتى تُرسل |
| أغلق التطبيق | ❌ لا | IndexedDB مستمر عبر الجلسات |
| عطل في GPS | ❌ لا | Timer يحاول مرة أخرى — إذا فشل GPS يتخطى الدورة |
| خطأ في الإرسال (server error) | ❌ لا | retry في الدورة التالية |
| تغيير الصفحة (within SPA) | ❌ لا | VisibilityChange → flush قبل المغادرة |
| المستخدم يغلق علامة التبويب | ❌ لا | قبل unload → محاولة flush متزامنة (قد لا تعمل دائماً — لكن النقاط في IndexedDB آمنة) |

## 19) الحفاظ على التوقيت الحقيقي للنقاط

### المبدأ

**`captured_at` = وقت التقاط الـ GPS** (حقيقي)  
**`created_at` = وقت الإدراج في القاعدة** (قد يختلف — لا يُستخدم للتحليلات الزمنية)

### كيف يُنفَّذ؟

```
1. GPS يلتقط نقطة:
   const point = {
     session_id,
     lat,
     lng,
     captured_at: new Date().toISOString(),  // ← وقت الالتقاط الحقيقي
     point_type: 'track',
   };

2. يحفظ في IndexedDB مع captured_at الأصلي (لا يتغير)

3. عند الإرسال (فوراً أو لاحقاً):
   sync_tracking_points({
     p_points: [{
       p_session_id: point.session_id,
       p_lat: point.lat,
       p_lng: point.lng,
       p_captured_at: point.captured_at,  // ← يُرسل captured_at الأصلي
       p_point_type: point.point_type,
     }]
   })

4. في SQL:
   INSERT INTO tracking_points (session_id, lat, lng, captured_at, point_type, created_at)
   VALUES (p_session_id, p_lat, p_lng, p_captured_at, p_point_type, NOW());
   --               ↑ وقت الالتقاط الحقيقي    ↑ وقت الإدراج (قد يتأخر ساعات)
```

### الفرق بين `captured_at` و `created_at`

| العمود | القيمة | المعنى |
|--------|--------|--------|
| `captured_at` | وقت التقاط GPS | يُستخدم للترتيب الزمني، المسار، التوقفات، التقارير |
| `created_at` | وقت الإدراج في القاعدة | يُستخدم للتدقيق فقط (هل النقطة متأخرة؟) |

### هيكل `sync_tracking_points` (موجود حالياً)

```sql
-- موجود في SQL — لا تغيير
CREATE OR REPLACE FUNCTION public.sync_tracking_points(
    p_token uuid,
    p_points jsonb  -- array من points مع captured_at
)
RETURNS jsonb
-- ...
```

الـ RPC يقبل `p_captured_at` لكل نقطة — لا يستخدم `NOW()`.

**لا تغيير مطلوب** — الآلية الحالية تحترم captured_at.

### أين يُستخدم captured_at؟

| الاستخدام | المصدر |
|-----------|--------|
| ترتيب نقاط Route Map | `ORDER BY captured_at` |
| حساب المسافة (Haversine) | ترتيب النقاط بـ `captured_at` |
| اكتشاف التوقفات الطويلة | الفرق بين `MAX(captured_at)` و `MIN(captured_at)` |
| وقت الزيارة (هل هو قرب نقطة GPS) | مقارنة `visits.check_in_at` مع `tracking_points.captured_at` |
| حالة الاتصال (آخر نقطة) | `MAX(captured_at)` لكل موظف |
| Timeline | ترتيب الأحداث بـ `captured_at` / `check_in_at` / `created_at` |

### ماذا لو كان جهاز المستخدم غير مضبوط (Wrong Time)?

```
المخاطرة: المستخدم يغير وقت جهازه → captured_at غير صحيح

الحل:
1. في frontend: استخدام `new Date().toISOString()` يعتمد على وقت الجهاز
2. لا يمكن التحقق من صحة وقت الجهاز من JS
3. الحل المؤقت: في SQL عند الإدراج — إذا captured_at > NOW() + 1 hour → يُعتبر خطأ ويُستخدم NOW()
4. الحل الدائم: (مستقبلي) استخدام Network Time Protocol (NTP) عبر API
```

**التوصية للمرحلة الأولى:** نقبل وقت الجهاز. إذا كان الفرق كبيراً (> ساعة) → الـ SQL يرفض النقطة ويعيد رسالة خطأ.

### السيناريو


```
1. المستخدم يُسجِّل الدخول → الصفحة الرئيسية تظهر
```

```
   → إذا في نقاط متبقية (لم ترسل قبل الإنهاء) → flushQueue() (last try)
```

### الكود المقترح لـ Tracking Engine

```typescript
class TrackingEngine {
  private isRunning = false;
  private sessionId: string | null = null;
  private employeeId: string | null = null;
  private timerId: number | null = null;

  async start(sessionId: string, employeeId: string) {
    // 1. تحقق من tracking_required
    const policy = await getEmployeeWorkPolicy(employeeId);
    if (!policy.tracking_required) {
      console.log('Tracking not required for this employee');
      return;
    }

    // 2. فتح IndexedDB Queue
    await openQueue();

    // 3. flush أي نقاط قديمة
    await this.flush();

    // 4. بدء الدورة
    this.sessionId = sessionId;
    this.employeeId = employeeId;
    this.isRunning = true;
    this.timerId = setInterval(() => this.tick(), 30_000); // 30 ثانية
  }

  async stop() {
    this.isRunning = false;
    if (this.timerId) {
      clearInterval(this.timerId);
      this.timerId = null;
    }
    await this.flush(); // إرسال آخر النقاط قبل التوقف
  }

  private async tick() {
    try {
      const pos = await captureLocation();
      const point = {
        id: crypto.randomUUID(),
        session_id: this.sessionId,
        lat: pos.coords.latitude,
        lng: pos.coords.longitude,
        accuracy: pos.coords.accuracy,
        captured_at: new Date().toISOString(),
        point_type: 'track',
        retry_count: 0,
        created_at: new Date().toISOString(),
      };

      // حفظ في IndexedDB
      await saveToQueue(point);

      // محاولة إرسال الـ Queue
      await this.flush();
    } catch (err) {
      // خطأ GPS أو إرسال — ignore, next tick
    }
  }

  private async flush() {
    // أنظر Section 18: Batch Sending Logic
  }
}
```

### أحداث `visibilitychange` و `beforeunload`

```typescript
// في App.tsx أو TrackingEngine
useEffect(() => {

    if (document.visibilityState === 'visible') {
      // رجع للتطبيق
```

| عدد الزيارات | ✅ دقيق | ✅ مجموع | ✅ مجموع | ✅ مجموع |
| عدد الطلبات | ✅ دقيق | ✅ مجموع | ✅ مجموع | ✅ مجموع |
| عدد التحصيلات | ✅ دقيق | ✅ مجموع | ✅ مجموع | ✅ مجموع |
| ساعات العمل | ✅ صافي | ✅ مجموع + متوسط | ✅ مجموع + متوسط | ✅ مجموع + متوسط |
| Target vs Actual | ✅ يومي | ❌ (لا ينطبق) | ✅ شهري | ❌ (إذا عبر شهور) |

### كيف يُحتسب Route للفترات المتعددة؟

```
1. الفترة = 1 يوم → كل tracking_points في ذلك اليوم (كامل)
2. الفترة = 7 أيام → tracking_points كلها لكن مع:
     • تكثيف: إذا > 500 نقطة → خذ نقطة كل 10 دقائق
     • Polyline يمر بكل النقاط (قد يكون كثيفاً)
     • Clustering للنقاط
3. الفترة = شهر → tracking_points كلها لكن:
     • تكثيف: نقطة كل 30 دقيقة
     • Polyline قد يكون غير مفيد (كثير النقاط) → خريطة حرارية بدلاً من Polyline

الحل الأمثل: خريطتان:
   Map 1: Polyline (للمسار)
   Map 2: Heatmap (للكثافة) — للفترات الطويلة
```

## 23) أثر المتطلبات الجديدة

### 23.1 على Database

| المتطلب | التغيير المطلوب |
|---------|----------------|
| **1. Data Retention** | لا تغيير — `tracking_points` تبقى كاملة حالياً. الأرشفة دراسة مستقبلية. `auto_cleanup_tracking_data` يُعدَّل ليكون no-op (أو يُهمل) |
| **2. Route Map** | لا تغيير — `get_employee_day_map` موجود وجاهز |
| **3. Offline Tracking** | لا تغيير — IndexedDB في الـ Frontend فقط |
| **4. Real Timestamp** | لا تغيير — `sync_tracking_points` يستخدم `p_captured_at` بالفعل |
| **5. Runtime Restore** | لا تغيير — منطق Frontend فقط |
| **6. Multi-Period Map** | RPC جديد: `get_employee_multi_day_map` (لا حاجة لجداول جديدة) |
| **7. Timeline تشغيلي** | `get_employee_day_timeline` يُوسَّع (لا حاجة لجداول جديدة) |

**الخلاصة:** المتطلبات الجديدة لا تحتاج جداول أو أعمدة إضافية في القاعدة. التغيير الوحيد: `auto_cleanup_tracking_data` يُعدَّل ليكون no-op.

|---------|---------|

- الميزة غير أساسية — النظام يعمل بدونها
- لا تؤثر على التقارير أو البيانات
- قد تحتاج تحسين أداء مع الكم الكبير من النقاط
- يمكن إضافتها في أي وقت لاحق

### Alerts History Table (جدول التنبيهات)

**عند الحاجة:** إذا تطلبت الإدارة تتبع تاريخ التنبيهات (متى ظهر التنبيه، من أغلقه، متى أُغلق).

### NTP Timestamp Verification

**عند الحاجة:** إذا ظهرت مشكلة توقيت جهاز غير مضبوط.

**الحل:** استخدام Network Time Protocol عبر API خارجي لمقارنة وقت الجهاز بالوقت الفعلي.

## الخلاصة

هذه الوثيقة هي التصميم النهائي المقترح بعد مراجعة `ATTENDANCE_REDESIGN_IMPACT_ANALYSIS.md` وإضافة 8 متطلبات تشغيلية جديدة.

**الخلافات مع التحليل السابق:**
1. ❌ **Executive** حُذف — الآن work_location + schedule_type
2. ❌ **حذف Alerts** أُلغي — الآن مركز تنبيهات ديناميكي بـ 10 أنواع
3. ❌ **حذف Live Monitoring** أُلغي — الآن مركز العمليات الميدانية (أهم شاشة)
4. ❌ **اسم WorkforceOverview** أُلغي — الاسم الجديد: مركز العمليات الميدانية
5. ✅ **Target vs Actual** أُضيف — الآن عنصر أساسي في كل شاشة
6. ✅ **5 شاشات** بدلاً من 7 — لا حذف، إعادة تصميم وتوسيع
7. ✅ **8 متطلبات جديدة** أُضيفت — Data Retention, Route Map, Offline Tracking, Real Timestamps, Runtime Restore, Multi-Period Maps, التشغيلية Timeline, وتحليل الأثر

**القرارات المضمنة في هذه النسخة:**
1. **Alerts**: ديناميكي — لا جدول `operational_alerts` حالياً
2. **Data Retention**: tracking_points تبقى كاملة — الأرشفة دراسة مستقبلية (عند > 1M صف)
3. **Work Hours Ledger**: جزء أساسي من التصميم — net_hours هو المؤشر الرئيسي المشترك
4. **3 أنواع work**: دوام ثابت (يقارن الوقت) / دوام مرن (يقيس الساعات فقط) / بالساعة (يحسب الدقيق)
5. **Route Replay**: Future Roadmap — غير مطلوب حالياً
6. **التغييرات الفعلية**: قائمة واضحة ومحددة في Database (جدول واحد + عمود واحد) / RPCs (5 جديدة + 10 تعديلات) / Frontend (2 service files + 6 components + 9 pages)
